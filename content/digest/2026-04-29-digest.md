## AI's Economics Don't Make Sense

**来源**: [AI's Economics Don't Make Sense](https://www.wheresyoured.at/ais-economics-dont-make-sense/)<br>
**作者**: Ed Zitron

> - **核心论点** — AI行业的经济模型从根本上不可持续，基于补贴和欺骗的订阅模式即将崩溃。
> - **订阅模式的欺骗性** — GitHub Copilot等产品长期以远低于实际成本的价格销售，用户每花1美元订阅费，供应商亏损高达8-13美元。转向按量计费后，用户面临数十倍的成本暴涨。
> - **成本结构矛盾** — 固定月费无法覆盖LLM服务的高波动性成本（单个任务可能烧掉数十美元），但企业故意隐藏实际token消耗，让用户误以为产品廉价可靠。
> - **数据中心财务脆弱** — 建一个100MW数据中心需30亿美元以上，即使满租，利润率仅5%-16%；若依赖债务，则为负值。大部分在建AI数据中心（114GW）需要1.18万亿美元年收入才能支撑，但主要客户（OpenAI、Anthropic）自身巨亏。
> - **OpenAI与Anthropic的幻想** — OpenAI需在2030年前筹集或赚取8520亿美元才能支付计算合同，但其CFO已公开担忧无法履约；Anthropic类似的百亿美元级计算承诺亦无盈利路径。
> - **系统性风险** — 若OpenAI失败，将拖垮Oracle（已负债超千亿美元建Stargate），引发连锁崩盘；作者坦言“不知道这一切如何收场”。
> - **现实意义** — AI产业的宣传依赖用户从未支付真实成本，一旦被迫面对实际价格，泡沫将迅速破裂，对金融市场和实体经济造成严重冲击。

**点评**: 这篇文章和我最近一直说的一个观点一致，就是目前一线 AI 公司的模型能力上升和赚钱是矛盾的，具体表象就是模型降智、砍优惠套餐、大涨价以及媒体叙事的日渐荒诞，这将是这一轮模型和智能体应用的一个 checkpoint，无论走向何方，目前的状态都是不可持续的。

## Understanding systems

**来源**: [Understanding systems](https://entropicthoughts.com/understanding-systems)<br>
**作者**: a@xkqr.org (kqr)

> - **核心论点** — 有效辅导的绝大多数工作（约80%）是动机管理：辅导者必须根据学生动机的升降实时调整内容难度，同时通过持续观察学生解题过程来识别并纠正其有缺陷的思维模型。
> - **动机管理** — 辅导者需敏锐察觉学生动机变化：动机下降时切换更轻松内容甚至闲聊，动机上升时提高难度。学生是否相信自己答案的强度决定了纠正错误信念所需的工作量，因为强烈错误信念会形成“知识护盾”。
> - **观察与假设** — 辅导者通过观察学生解题动作，不断形成关于学生思维模型的假设，并选择能暴露潜在缺陷的下一道习题来检验假设。对新学生，需先通过随机习题快速校准其对各种思维模型的使用情况。
> - **验证能力的培养** — 辅导者要求学生在解题后自证答案（而非直接看题解），并传授三种验证方法：用不同方法重算、进行可行性检查（直观边界）、对多步解法递归验证最不确定的步骤。这是学生极少具备的关键技能。
> - **错误类型区分** — 辅导者需区分“非生产性错误”（如笔误，可忽略或轻描淡写纠正）与“生产性错误”（暴露根本误解，适合深入讨论）。在早期识别学生走错方向时，及时介入要求验证以维持动机。
> - **局限性/不确定性** — 原文基于作者作为数学/物理私人辅导员的个人经验，且引用的核心文章已不可寻。作者坦言，该方法在越远离数学的领域可能越难应用。
> - **实践意义** — 辅导的核心技能——不断形成假设并通过精心设计的问题检验假设——是可迁移的，同样适用于调试代码、理解网络结构或学习驾驶赛车等任何系统理解过程。

**点评**: 作者从自己作为数学和物理私教的经历出发，聊了关于教育的核心挑战：动机管理。在 AI 时代，理解这一点尤为重要，因为纯知识传授已经不是教育的主要问题。

## GitHub Actions is the weakest link

**来源**: [GitHub Actions is the weakest link](https://nesbitt.io/2026/04/28/github-actions-is-the-weakest-link.html)<br>
**作者**: Andrew Nesbitt

> GitHub Actions 的多项“按设计工作”的功能组合在一起，使得它成为近年来一系列重大开源供应链攻击的共同根源，而平台的默认不安全设置与迟缓的修补让维护者独自承担风险。
> 
> - **核心论点** — 从2024年底到2026年的多起供应链攻击（spotbugs、Ultralytics、tj-actions、nx/s1ngularity、Trivy、elementary-data）均可追溯到 GitHub Actions 的几个特性：`pull_request_target` 触发器在未审查的 fork 上下文中授予完整密钥和写权限，可变 git 标签允许已授权的提交被轻易替换，`$` 模板注入将不可信输入直接拼入 shell 脚本，以及默认具有写权限的 `GITHUB_TOKEN`。
> - **共同技术缺陷** — 这些工具并非 bug，而是设计决定：`pull_request_target` 和 `issue_comment` 触发器运行于完全信任的环境；动作版本是他人仓库中的可变 git 引用；缓存跨信任边界共享而不加标识；`$` 扩展不经引号即替换。攻击者只需一个 PR 标题或注释即可触发恶意代码执行并窃取凭据。
> - **关键限制** — GitHub 的安全路线图（工作流锁定文件、策略控制等）全部默认为 opt-in，且需要三至六个月才能进入公开预览。社区讨论中 GitHub 员工强调更改默认值会破坏已有工作流，但正是这些“已有工作流”在持续产生漏洞——91% 的 PyPI 包使用可变标签引用第三方动作，三分之二的工作流缺少 `permissions:` 块。
> - **后果与严重性** — PyPI、npm、RubyGems、crates.io 均已采用 OIDC 可信发布，将信任从分散的维护者凭证集中到 GitHub Actions 平台。这意味着一旦攻击者攻破一个构建工作流，即可直接向主流注册表发布恶意包，而无需逐个钓鱼维护者。
> - **建议** — 目前最有效的防护是使用第三方检查工具 zizmor（四条 YAML 即可启用）来捕获危险触发器、未锁定引用和权限过宽等问题。在 GitHub 改变默认值之前，维护者应固定 SHA、设置 `permissions: {}` 并在每个工作流顶部声明权限，同时假设任何不可信输入（PR 标题、分支名、issue 评论）最终会被当作脚本执行。
> 
> 当前平台默认将写权限与密钥交给未经验证的贡献者，而整个软件生态的发布信任链正日益依赖这一缺陷平台，这使得 GitHub Actions 成为最薄弱的环节。

**点评**: 之前在我们的播客节目里提过 GitHub Actions 的安全隐患，因为时间原因没有展开说，这篇讲的比较具体。一般来说，类似 GitHub Actions 这样强大的工具以如此小的限制被普遍使用，一定是有很多隐患的。

## Ghostty Is Leaving GitHub

**来源**: [Ghostty Is Leaving GitHub](https://mitchellh.com/writing/ghostty-leaving-github)<br>
**作者**: Mitchell Hashimoto

> - **核心决定** — 由于 GitHub 频繁发生故障（过去一个月几乎每天都有），严重阻碍日常工作（如写作当天因 Actions 中断两小时无法进行 PR 审查），作者宣布将 Ghostty 项目迁出 GitHub，并计划在未来几个月确定迁移目标平台。
> - **情感背景** — 作者自 2008 年起每日使用 GitHub 长达 18 年，视其为最快乐的地方，甚至为求职而创建 Vagrant。近期因平台频繁不可用而转为愤怒与失望，认为 GitHub “不再适合严肃工作”。
> - **迁移范围** — 仅 Ghostty 项目及所属社区受影响；作者的个人项目和其他工作暂时保留在 GitHub。届时 GitHub 上会保留只读镜像。
> - **不确定性** — 具体迁移目标仍在讨论中，涉及多个商业与开源提供商；整个迁移将逐步进行，以最大限度减少依赖。作者表示若 GitHub 有实质改进，不排除未来回归的可能性。

**点评**: 近期非常流行的终端模拟器 Ghostty 的作者 Mitchell Hashimoto 宣布将把 Ghostty 迁出 GitHub，他甚至还没想好迁到哪里去，但实在被 GitHub 近期越来越严重的可用性问题气坏了。GitHub 到底出了什么问题？官方 blog 同日发了一篇文章来做出[解释](https://github.blog/news-insights/company-news/an-update-on-github-availability/)，里面援引的运维数据显示，自去年年底开始，所有类型的请求数据都有显著增长，超出了 GitHub 基础架构能平稳应对的阈值。

## Before GitHub

**来源**: [Before GitHub](https://lucumr.pocoo.org/2026/4/28/before-github/)<br>
**作者**: Armin Ronacher

> GitHub 曾是开源社区不可替代的中心，但随着它的衰落，我们需要反思历史并建设一个不依赖单一公司、能长久保存源代码及其社会背景的公共档案系统。
>
> - **GitHub之前的开源世界** — 依赖项目少且规模大，开发者需要自建基础设施（如 Trac、Subversion），加入依赖意味着要理解其历史和维护者，信任和声誉是稀缺资源，但项目因个人服务器过期而丢失的现象普遍。
> - **GitHub的贡献** — 它极大降低了创建、发现、贡献开源的门槛，提供了统一的代码托管、问题追踪、CI、API等工具；更重要的是它意外地成为一座“图书馆”，即使废弃的项目也保持可检索，保存了代码和社会协作的历史。
> - **GitHub当前的衰落** — 平台正因不稳定、产品方向混乱、Copilot AI 噪音、领导层缺失而失去社区信任；已有重要项目（如 Ghostty、Strudel）宣布迁移至 Codeberg 等替代平台，但大规模迁移尚未发生。
> - **分散化的代价** — 若项目回到分散的自托管模式，虽然能恢复自主性，但容易丢失问题的讨论、评审记录、发布说明等“社会上下文”；历史上邮件列表和自建服务器已证明这些数据脆弱易逝。
> - **需要公共档案** — 不依赖任何单一公司商业模式或领导层喜好的、有充足资金支持的存档机制至关重要，用来保存源代码、发布产物、元数据和足够的项目背景，避免重蹈早期“断链Trac实例”的覆辙。
> - **局限性/不确定性** — 目前没有明确的方案能兼顾分散自主与全面存档；GitHub是否真的会加速衰落、替代者能否普及、以及如何迁移“社会上下文”都还是未知数。
>
> 实际意义：开源社区不能假设GitHub永远可靠，而应主动建设可持续、去中心化的记忆基础设施——让项目易于迁移、镜像和保留，使历史不因一家公司的转向而沦为文化危机。

**点评**: 这篇文章则更深入的探讨了开源社区的组织和运营模式问题。其实互联网世界早期都是分散的体系，这甚至是互联网基础架构被设计出来时的核心假设，但随着互联网商业化开发越来越深入，集中运营的中心化平台提供的体验更符合用户“省事省心”的期望，问题是，中心化甚至垄断性的平台是会出问题的，那时候人们就会面临痛苦的抉择。所以，未来到底该往哪里走呢？还有得选吗？
